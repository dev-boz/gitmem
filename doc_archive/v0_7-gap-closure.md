## A. Add to spec (missing Tier 2 items)

### A.1 `source_type` enum (was 2.20)

Add `source_type` to the fact schema. This is the single biggest retrieval quality improvement available ‚ÄĒ it lets composite scoring distinguish a fact read from actual source code from one an LLM inferred from context.

**Enum values:**

| Value | Meaning | Example |
|-------|---------|---------|
| `ground_truth_code` | Read directly from source file by an adapter | "function returns a Promise" parsed from AST |
| `user_prompt` | User explicitly stated in conversation | "always use snake_case in this repo" |
| `tool_output` | Tool native memory ‚ÄĒ LLM deliberately wrote it | Claude Code CLAUDE.md entry |
| `llm_inference` | LLM inferred/reasoned during session | "this probably uses connection pooling" |
| `dream_consolidation` | Derived by the dream pipeline from other facts | Cross-session pattern extraction |
| `external_doc` | Extracted from external documentation | README, API docs, wiki |

**Where to add:**
- Fact schema in Section 5 ‚ÄĒ add `source_type` field (MUST, enum)
- Inline metadata table in Section 9 ‚ÄĒ add `st` key (MUST)
- Composite scoring in Section 6 ‚ÄĒ add `w_st √ó source_type_weight` term. Ground truth scores highest, LLM inference lowest. Weights configurable.
- Config reference Section 27 ‚ÄĒ add `weights.source_type` default
- Reference [14] (Johnson et al. Source Monitoring) ‚ÄĒ note in Section 4 that `source_type` operationalises this framework

### A.2 CONVENTIONS.md (was 2.17)

Add a new file to the memory repo directory layout: `CONVENTIONS.md`. This is the project's schema file ‚ÄĒ analogous to Karpathy's `CLAUDE.md` schema layer but for memory, not for the agent.

**Purpose:** Defines the project's topic taxonomy, fact phrasing norms, entity vocabulary, and any project-specific conventions that L1/L2 dream agents must follow when extracting and reviewing facts.

**Where to add:**
- Section 8 directory layout ‚ÄĒ add `CONVENTIONS.md` at the root of each project memory repo (alongside `meta/`)
- Section 11 Dream Pipeline ‚ÄĒ Orient phase MUST read `CONVENTIONS.md` before processing. Gather phase uses it to guide extraction. L2 review checks proposed facts against it.
- Section 12 gitmem ‚ÄĒ L2 reviewer prompt includes `CONVENTIONS.md` as context
- Section 9 ‚ÄĒ note that `CONVENTIONS.md` is human-authored (S:5 equivalent) and committed to main. It is NOT auto-generated by dream agents.

**Example content:**

```markdown
# Project Conventions

## Topic taxonomy
- devenv: local development environment setup
- auth: authentication and authorization
- deploy: deployment pipeline and infrastructure

## Fact phrasing
- Present tense for current state: "postgres runs on 5433"
- Past tense for historical: "previously used MySQL"
- Never combine two facts in one line

## Entity vocabulary
- "the API" = src/api/ Express server
- "the worker" = src/jobs/ background processor
- "staging" = GCP project boz-staging

## Project-specific conventions (deviations from common practice)
- We use port 5433 not 5432 for postgres (to avoid conflict with system postgres)
- Test files are co-located with source, not in a tests/ directory
```

### A.3 Lint sub-phase in Consolidate (was 2.16)

Add a Lint sub-phase to the Consolidate phase (phase 3) of the Dream pipeline. Currently Consolidate only merges candidates and resolves conflicts. Lint catches structural integrity problems.

**Lint checks:**
- Contradiction scan: facts with high semantic similarity but opposing assertions that weren't caught by `conflicts_with` (requires embedding comparison, not just exact-match)
- Stale file references: `files/` and `folders/` scope entries pointing to paths that no longer exist in the project repo
- Orphan `fact_id` references: IDs in `corroborated_by`, `conflicts_with`, or `provenance.sessions` that don't resolve to existing facts or sessions
- Tag drift: same concept tagged inconsistently across facts (e.g., `database` vs `db` vs `postgres`)
- Convention violations: facts that don't conform to `CONVENTIONS.md` phrasing or taxonomy rules

**Where to add:**
- Section 11 phase table ‚ÄĒ split Consolidate into two sub-phases: Consolidate (merge/resolve) then Lint (integrity checks)
- Section 12 gitmem ‚ÄĒ Lint findings in `remote`/`hybrid` mode open a dedicated `[dream/lint]` PR with label `type: lint`. L2 auto-merges obvious fixes (orphan cleanup, tag normalisation). Contradictions escalate.
- Section 13 ‚ÄĒ add `lint` to commit message types

### C.2 Bi-temporal facts (2.18)

Add to Open Questions: "Bi-temporal fact validity (`valid_from` / `valid_to` / `umx query --as-of`) was proposed for v0.7 but deferred. Most coding facts are point-in-time current state, not time-ranged assertions. `expires_at` covers the common TTL case. `supersedes`/`superseded_by` chains provide implicit temporal evolution. Full bi-temporal support adds significant schema and query complexity for a narrow set of use cases. Revisit if time-sensitive facts (deploy targets, feature flags, API versions) prove problematic in practice."

### C.3 Hot/warm/cold tiers (2.19)

Add to Open Questions: "Formal hot/warm/cold memory tiers were proposed but the behaviour already exists implicitly: `MEMORY.md` + `manifest.json` are always-loaded (hot), `facts/topics/` are loaded on-demand by relevance scoring (warm), `sessions/` are never loaded unless explicitly queried (cold). Naming the tiers doesn't change the implementation. If promotion/demotion rules between tiers are needed (e.g., demote a hot fact to warm after N days without retrieval), formalise then."
